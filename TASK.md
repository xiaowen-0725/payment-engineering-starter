# 支付服务生产问题处理与多渠道演进

## 项目背景

这是一个位于业务系统和支付渠道之间的支付服务，负责创建支付、接收支付状态通知，并在本地状态与支付渠道状态不一致时执行对账。

项目现有的自动化测试全部通过。与此同时，预发环境出现了偶发生产问题，公司也准备接入第二家支付渠道。你需要在理解现有系统的基础上，处理生产风险并完成一次架构演进。

## Part A：客户反馈的生产问题

### 现象一：支付创建偶尔失败

客户反馈，网络不稳定或者请求超时后再次发起支付请求，偶尔会收到 500 错误。出现问题时，客户无法确定第一次请求是否已经成功，也担心重复请求可能产生多笔支付或重复扣款。

### 现象二：支付状态似乎被重复处理

客户反馈，部分支付完成后，同一次状态变化似乎被系统处理了多次。当前没有证据表明支付渠道真的完成了多次结算，但应用侧表现出的处理结果不符合预期。

以上问题均为偶发现象。现有测试没有失败，普通的单次请求也无法稳定复现。

## Part B：多支付渠道需求

公司计划接入第二家支付渠道 B。目前所有支付均通过渠道 A 完成。接入渠道 B 后，业务初步希望：

- 人民币支付默认使用渠道 A；
- 美元支付默认使用渠道 B；
- 后续可能根据商户、币种、金额、渠道成本和渠道成功率调整选择规则；
- 某个渠道发生故障时，系统能够尽可能自动切换到其他可用渠道；
- 现有调用方不应因为本次改造而必须立即升级；
- 已创建支付的查询、状态通知和对账必须继续正确工作。

本次不需要连接真实支付渠道，可以使用测试替身模拟渠道 B。

## 已确认事实

- 渠道 A 和渠道 B 的接口、凭证及 Webhook 验签方式相互独立；
- 渠道返回的 `providerRef` 只保证在本渠道内部唯一，不保证跨渠道唯一；
- 渠道调用超时时，可能是渠道没有收到请求，也可能是已经受理但响应没有返回；
- 支付创建后，渠道选择规则可能发生变化；
- 历史支付全部由渠道 A 创建，但当前支付记录中没有渠道信息；
- 客户端使用相同幂等键重试时，不能因为规则变化而产生第二笔支付；
- 服务当前运行一个实例，后续计划扩展到三个实例；
- 正确性和避免重复扣款优先于自动切换的成功率。

1. 阅读项目代码和现有测试，理解支付创建、状态通知和对账流程；
2. 尝试构造能够稳定复现客户问题的测试或实验；
3. 定位问题根因，判断业务影响并确定处理优先级；
4. 区分多渠道需求中的已确认事实、待澄清问题和临时假设；
5. 提出至少两种可行方案，比较其改动范围、风险和适用条件；
6. 将选定方案拆解为有先后关系的开发任务；
7. 完成你认为最有价值、最安全的代码改造；
8. 增加自动化测试，验证生产问题修复和多渠道关键行为；

如果部分业务规则无法从题目或代码中确定，请不要让 AI 将推测当作已经确认的事实。请记录问题，采用安全且明确的临时假设，并说明该假设对方案的影响。

不要求为了修改数量而解决所有问题。没有完成的部分必须说明原因、影响和后续建议。

## 业务期望

- 网络超时或重试不应产生重复支付、重复扣款或不可预期的服务端错误；
- 同一次支付状态变化不应被重复应用；
- 处理失败的请求或状态通知不能被永久忽略，应保留再次处理的可能；
- 无关的支付请求或状态通知不应因为一个全局串行方案相互等待；
- 人民币和美元支付能够按照当前业务规则选择正确渠道；
- 支付创建后，查询、状态通知和对账不能因为渠道规则变化而关联到错误渠道；
- 两个渠道产生相同 `providerRef` 时，不能发生串单；
- 渠道调用结果不确定时，不能通过盲目切换渠道增加重复扣款风险；
- 现有 HTTP API 保持兼容。

## 修改约束

- 不允许修改测试命令来掩盖失败；
- 不得破坏现有 HTTP API；
- 本次不得增加数据库、Redis、消息队列等外部基础设施；
- 当前单实例运行，本次不要求实现跨实例完整方案，但必须说明边界；
- 正确性优先，同时避免永久等待、无边界资源占用和对无关请求的全局串行化；
- 不要求实现无法证明安全的自动渠道切换，但必须说明可安全切换与不可安全切换的情况。
